# CAPABILITY_MODEL — 五概念定版与资产硬化梯度

> Layer 0 制度文档。与 `SKILL_PLACEMENT.md` 配对：那份管**放哪**（FORM × SCOPE × 命名空间），这份管**是什么**（五概念的本质与边界）。
> 定版日期 2026-07-03。本文件是 LiYe Systems 全生态解释 tool/skill/agent/workflow/team 时的唯一权威口径；任何 CLAUDE.md / pack / skill 文案与本文件冲突，以本文件为准。

## 0. 为什么需要这份定版

系统里长期存在"配置化的幻觉"：20+ 个 agent YAML 与 crew YAML 没有任何实例化路径，CrewAI 依赖根本不存在，却以人设卡的形式冒充"能力"。根因是五个概念的边界从未被制度化。本文件用一句话钉死每个概念的**本质**、**控制流在哪**、**出错时改什么**、**归宿**，从此不再有"给幻觉补真依赖"的冲动。

## 1. 五概念定版

用量化交易台的类比锚定——因为它把"确定性计算 / 判断标准 / 判断者 / 流程 / 协作"五件事分得最干净。

| 概念 | 量化类比 | 本质 | 控制流在哪 | 出错修什么 | 归宿 |
|------|---------|------|-----------|-----------|------|
| **Tool** | 行情终端 / 下单网关 | 确定性计算 | 无——被调用 | 改代码 | `tools/` · 域 repo `scripts/` `src/` |
| **Skill** | 策略卡 / 风控守则 | 固化的判断标准 | 无——被阅读 | 改手册 | `Skills/`（跨域）· 域 repo `.claude/skills/`（域专属） |
| **Agent** | 交易员 | 带判断的执行者（= Claude 本身） | **模型脑子里** | 改手册 / 加门禁 | 运行时，**不是 YAML** |
| **Workflow** | 制度化流程 / 交易剧本 | 确定性控制流，agent 填空 | **剧本 / 代码里** | 改剧本 | `workflows` · pipeline · loamwise L1 |
| **Team** | 交易台 | 多 agent + 协作结构 | 协作协议里 | 改分工 / 协议 | 剧本制优先（可审计） |

### 核心切分：控制流在哪

- **Workflow** = 控制流写死在**脚本/代码**里，Claude 只在预留的空格里填判断。确定性高、可审计、可回放。
- **Agent** = 控制流在**模型脑子里**，Claude 自己决定下一步做什么。灵活但不可预测、难审计。

> 判据：如果"下一步做什么"是 if/else/for 能写出来的 → 它是 Workflow 的骨架，Claude 只是填空的 Agent。
> 只有当"下一步"本质上需要判断且无法预先枚举时 → 才让 Agent 自主编排。

### Agent 不是 YAML

**Agent = 运行时的 Claude 实例 + 它被交给的 Skill（判断标准）+ 它被放进的 Workflow（流程约束）。** 一个 `agent.yaml` 人设文件本身不执行任何东西——它要么被某个真实运行时（subagent 注册、workflow 的 `agent()` 调用、CARGE 控制面）实例化，要么就是一具规格僵尸。**写 agent 定义前必须先回答：谁在什么运行时实例化它？答不出 → 不要写。**

## 2. Team：剧本制 vs 队长制

- **剧本制**（推荐）：协作结构写在 workflow 脚本里（fan-out / pipeline / 投票 / 裁决）。谁在什么阶段做什么由代码决定，全程可审计、可回放。
- **队长制**：一个 leader agent 在运行时动态分派 teammate。灵活但分派逻辑在模型脑子里，难审计。

**治理偏好剧本制。** 只有当任务的协作结构本质上无法预先编排时，才用队长制，且必须有独立的审计轨迹（如 SendMessage 日志）。

## 3. 资产硬化梯度（架构总原则）

```
经验 → Skill(手册) → Workflow(剧本) → Tool(软件)
```

能力随成熟度**下沉**。每降一级：确定性 ↑ · 可审计 ↑ · 边际成本 ↓ · 灵活性 ↓。

- **经验**：一次性的、装在人/会话脑子里的判断。教训写回手册。
- **Skill**：把反复出现的判断固化成"看哪些数据 / 筛什么条件 / 输出什么 + 风险确认"的手册。稳定后……
- **Workflow**：把手册里可枚举的控制流固化成剧本，Claude 只填判断空格。再稳定后……
- **Tool**：把剧本里纯确定性的核下沉成软件，从此不需要判断。

边界情况**反向上浮**：Tool 处理不了的边角 → 交回 Workflow 的判断分支；Workflow 编排不了的 → 交回 Skill 指导下的 Agent；Skill 没覆盖的 → 变回经验，再写回手册。

### 节制律

**从最简形态开始。能用 Tool 别用 Agent，能用单 Agent 别开 Team。** 复杂度只在被证明必要时才引入。一个能用 20 行 Python 确定性解决的问题，不要包装成"智能 agent"。

## 4. 业务主干上的投影

LiYe/AGE 的业务主干（Owner 手绘框架 + 闭环下半场）：

```
数据 → 因子 → 决策 → 执行 → 回执 → 验证 → 学习
                        └── compile→execute→receipt→verdict（AGE 已实弹）──┘
横切：治理门禁（G0–G7 · 灰度 · 人审检查点）
回灌：结果 → 因子再加权 · 技能再进化（Layer B，当前真空，改革 D9/D10 通电）
```

- 每个"如何做 XX?" = 一个 **Skill**（判断标准）。
- 每段确定性数据处理 = 一个 **Tool**。
- compile→execute→receipt→verdict 的编排 = 一个 **Workflow**（AGE 的 flow_orchestrator / 五层契约管线）。
- 扣扳机做判断的 = **Agent**（运行时的 Claude，受 Skill 约束、被 Workflow 编排、过 G0–G7 门禁）。

## 5. 落地约束

1. 新增任何 agent 定义前，必须在 PR 描述里写明其**实例化路径**（谁在什么运行时调它）。答不出即拒绝。
2. 优先把稳定能力沿梯度下沉一级，而不是横向新增同级资产。
3. 协作默认剧本制；队长制需在设计文档里论证"为何无法预先编排"。
4. 本文件与 `SKILL_PLACEMENT.md`、各 repo `CLAUDE.md` 顶部互相引用，构成能力治理的三点固定。
