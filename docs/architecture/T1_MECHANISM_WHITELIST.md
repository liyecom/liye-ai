# T1 Mechanism Whitelist

> **Status**: CANONICAL
> **Purpose**: 定义可进入 T1 的机制类型，防止内容稀释
> **Enforcement**: TRUTH_DELTA_GATE + Manual Review

---

## Core Principle

> **T1 只接受机制，不接受内容。**

机制是可复用的因果结构，内容是一次性的信息载体。

T1 的价值来自其机制密度，而非知识广度。一个精确的机制胜过一百条泛化建议。

---

## Whitelist: 允许进入 T1 的机制类型

### 1. Causal Chain (因果链)

**定义**: A → B → C 形式的因果关系链条

**特征**:
- 明确的起点和终点
- 可追溯的中间环节
- 可验证或可证伪

**示例**:
```
✅ "关键词过多 → 单词预算不足 → 无法收集有效数据 → 无法做出正确优化决策"
✅ "价格高于类目阈值 → Amazon 抑制 Buy Box → 转化率下降 → BSR 下滑"
```

**反例**:
```
❌ "关键词很重要"（无因果）
❌ "要优化预算"（无链条）
```

---

### 2. Threshold Rule (阈值规则)

**定义**: if X > Y then Z 形式的条件触发规则

**特征**:
- 明确的触发条件
- 具体的数值或比例阈值
- 可量化的结果

**示例**:
```
✅ "当单个广告活动关键词超过50个时，预算分散导致单词日均展示不足3次"
✅ "当产品价格高于类目 Top 10 均价 20% 时，Buy Box 展示概率显著下降"
```

**反例**:
```
❌ "关键词不要太多"（无具体阈值）
❌ "价格要有竞争力"（无量化标准）
```

---

### 3. Failure Mode Pattern (失效模式)

**定义**: 在特定条件下系统/策略失效的模式

**特征**:
- 明确的失效场景
- 可识别的前置信号
- 失效后的具体后果

**示例**:
```
✅ "当 FBA 仓库持续发错商品时，Amazon 会从无关卖家账户扣款补偿买家"
✅ "当卖家在其他平台提供更低价格时，Amazon 会抑制该 ASIN 的 Buy Box"
```

**反例**:
```
❌ "有时候会出问题"（无具体模式）
❌ "要注意风险"（无失效识别）
```

---

### 4. Optimization Trade-off (优化权衡)

**定义**: 两个目标之间的内在冲突和最优平衡点

**特征**:
- 明确的对立目标
- 可量化的权衡关系
- 最优点的判断标准

**示例**:
```
✅ "广告覆盖面 vs 数据质量：关键词数量上限 = 日预算 / CPC / 3（确保每词每日至少3次点击）"
✅ "价格竞争力 vs 利润率：最低可接受价格 = 成本 × (1 + 目标利润率) + 平台费用"
```

**反例**:
```
❌ "要平衡各方面"（无具体权衡）
❌ "要找到最优点"（无判断标准）
```

---

### 5. Constraint / Boundary Condition (约束/边界条件)

**定义**: 策略适用范围的明确边界

**特征**:
- 明确的适用前提
- 边界外的行为预测
- 可验证的边界条件

**示例**:
```
✅ "本策略假设：产品转化率 > 类目平均。若转化率本身低于平均，广告优化无法弥补产品力不足"
✅ "Buy Box 抑制机制仅对第三方卖家有效，Amazon 自营商品不受此限制"
```

**反例**:
```
❌ "一般情况下有效"（无明确边界）
❌ "结果可能因情况而异"（无具体条件）
```

---

### 6. Platform Mechanism (平台机制)

**定义**: 平台算法/政策的内在运作逻辑

**特征**:
- 基于可观察行为推断
- 可预测的平台响应
- 可利用或需规避

**示例**:
```
✅ "Amazon 算法会将预算优先分配给历史表现更好的关键词"
✅ "TikTok 的购物用户处于发现模式而非搜索模式，因此内容比 Listing 更重要"
```

**反例**:
```
❌ "Amazon 算法很复杂"（无具体机制）
❌ "平台规则会变"（无可操作信息）
```

## Blacklist: 禁止进入 T1 的内容类型

### ❌ Best Practices (最佳实践)

**原因**: 缺乏因果解释，无法迁移到新场景

**识别特征**:
- "应该..."、"建议..."、"最好..."
- 无"因为"说明
- 无边界条件

**示例**:
```
❌ "标题前80字符应包含核心关键词"
❌ "主图应使用纯白背景"
❌ "每周应审查搜索词报告"
```

---

### ❌ Tips / Tricks (技巧)

**原因**: 一次性操作，无可复用机制

**识别特征**:
- "小技巧"、"诀窍"、"秘诀"
- 操作步骤而非因果逻辑
- 无法解释"为什么有效"

**示例**:
```
❌ "用 ChatGPT 生成标题"
❌ "在周末调整竞价"
❌ "先跑自动广告收集关键词"
```

---

### ❌ Step-by-step Tutorial (分步教程)

**原因**: 是操作指南而非推理依据

**识别特征**:
- "第一步...第二步..."
- 按钮/菜单操作说明
- 界面截图或路径描述

**示例**:
```
❌ "打开 Seller Central，点击广告，选择创建活动..."
❌ "在 Helium 10 中，输入关键词，点击搜索..."
```

---

### ❌ Subjective Summary (主观经验总结)

**原因**: 不可验证，可能有幸存者偏差

**识别特征**:
- "我觉得..."、"根据我的经验..."
- 无数据支撑的断言
- 个人观点而非普遍机制

**示例**:
```
❌ "我认为视频比图片更有效"
❌ "根据我的经验，周一流量最高"
❌ "大多数卖家都忽略了这一点"
```

---

### ❌ Vague Generalizations (模糊泛化)

**原因**: 无法产生具体推理

**识别特征**:
- "通常"、"一般"、"往往"无具体条件
- 无量化指标
- 无边界说明

**示例**:
```
❌ "好的标题通常能提高点击率"
❌ "A+ Content 一般能提升转化"
❌ "评论多的产品往往排名更高"
```

---

## Validation Checklist

在 T1 单元入库前，检查以下条件：

```
□ 包含明确的因果关系或条件触发
□ 因果链条完整（无跳跃）
□ 有具体数值/阈值（如适用）
□ 有边界条件说明
□ 不是操作指南
□ 不是主观经验
□ 可在不同场景中复用
```

如果以上检查有 ≥2 项不满足，该内容不应进入 T1。

## Enforcement

### TRUTH_DELTA_GATE 自动检查

```python
# Already implemented in truth_delta_gate.py
VAGUE_PATTERNS = [
    r'^(通常|一般|往往|大多数|很多)',
    r'(应该|建议|最好|推荐)$',
    r'^(我认为|我觉得|根据经验)',
]

MECHANISM_INDICATORS = [
    r'(因为|所以|导致|由于|因此)',
    r'(当.*时|如果.*则)',
    r'→',
]
```

### Manual Review 补充检查

对于通过 GATE 但仍存疑的单元，人工检查：
- 是否真的包含可复用机制
- 是否只是"看起来像机制"的内容
- 是否有更精确的表达方式
